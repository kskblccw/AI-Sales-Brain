# 电商智能客服系统 -- 简历项目 + 面试问答

## 一、简历项目描述

**项目名称**：基于 LangGraph 的多智能体电商客服系统

**项目时间**：2026 年 3 月 -- 2026 年 7 月

**技术栈**：Python / LangChain / LangGraph / FastAPI / PostgreSQL / Chroma / SSE / Docker

**项目描述**：

独立设计与实现了一套基于 LangGraph Supervisor 模式的多智能体电商客服系统，支持 200 款商品、18 个品类、50 笔订单的完整业务闭环。系统采用 Supervisor + 4 个专业 Agent（订单/商品/售后/FAQ）的协作架构，结合 Chroma 向量数据库实现 RAG 商品知识库（463 个文本块），实现了意图识别、工具调用、流式对话、用户确认和记忆持久化的完整链路。

**核心工作**：

- 使用 LangGraph StateGraph 构建 Supervisor 多智能体调度框架，实现正则预路由 + LLM 决策的双层意图识别，正则命中率 80% 以上
- 设计 ReAct 工具调用循环，封装 12 个业务工具函数，通过 RunnableConfig 实现手机号静默传递，LLM 上下文中不暴露用户隐私数据
- 基于 Chroma 构建 RAG 商品知识库（商品详情 + 品类选购指南 + 售后政策 + FAQ），自定义 DashScope Embeddings 客户端解决 API 兼容问题
- 使用 PostgresSaver 实现会话状态持久化，设计滑动窗口 + LLM 摘要压缩 + 用户画像的三层记忆系统，后台异步执行不阻塞主流程
- 通过 FastAPI + SSE 实现流式逐 token 输出，设计轮询机制解决 interrupt 暂停时流式连接阻塞问题
- 敏感操作（售后申请、地址修改）采用 interrupt + 用户弹窗确认机制，拒绝时自动回滚数据变更
- 独立端口构建知识库管理系统，支持文档 CRUD、语义搜索、长文档自动切分、索引重建
- 设计 MCP 扩展机制，实现 web_search 网页搜索和 demo_notify 工单工具，双模设计兼容 MCP 协议和直接调用
- 实现图片上传 + 视觉 LLM 多模态入口，支持粘贴/拖拽/选择图片，分析文本对用户透明
- 编写 100 条自动化评估案例，5 个评估维度（意图准确率、关键词覆盖、LLM-as-Judge、结构完整性、长度合理性）

## 二、面试官高频问题

### 架构与设计

**Q1: 为什么选择 Supervisor 模式而不是固定的路由规则？**

Supervisor 模式的核心价值在于 LLM 可以根据上下文灵活决策。纯规则路由虽然快，但无法处理"我先问订单，再顺便问问这个商品能不能退货"这种复合意图。Supervisor 每轮对话动态决定下一步是继续当前 Agent 还是切换或结束。同时我在 Supervisor 外层加了正则预路由覆盖 80% 高频场景，兼顾了速度和灵活性。

**Q2: 四个 Agent 之间如何通信？共享哪些状态？**

四个 Agent 不直接通信。它们各自是独立的 StateGraph 子图，只暴露统一的消息接口。主图 Supervisor 通过 `call_*_agent` 节点调用子图，传入 `state["messages"]` 和 `RunnableConfig`，子图返回处理后的消息。共享状态只包括：消息列表（通过 LangGraph 的 `add_messages` reducer 合并）、当前意图、调度轮数、用户手机号。

**Q3: 为什么每个 Agent 要封装为独立的 StateGraph 而不是全部放在一张图里？**

三个原因：一是隔离性，每个 Agent 有独立的 System Prompt 和工具集，不会互相干扰；二是可维护性，新增 Agent 只需加一个子图文件 + 在主图注册节点和边；三是子图的 ReAct 循环（Agent -> Tool -> Agent）是独立的，不会和主图的调度循环嵌套出问题。

**Q4: 如何处理用户上下文爆炸（对话历史过长）？**

三层记忆系统：滑动窗口保留最近 6-12 条消息原样参与推理；超出窗口的历史由 LLM 压缩为 150 字以内的情景摘要存 PostgreSQL；从对话中抽取用户偏好和事实存入画像表。每轮对话开始时注入压缩后的摘要和画像到 System Prompt，控制在 300 token 以内。压缩在后端异步线程执行，用户看到回复后前端静默触发，不阻塞主流程。

### 安全与隐私

**Q5: 手机号如何在系统中流转？为什么不会泄露给 LLM？**

手机号存储在服务端的 `session_phones` 表（PostgreSQL），通过 LangGraph 的 `RunnableConfig` 机制传递：`config["configurable"]["user_phone"]`。所有工具函数从 config 中读取手机号进行身份验证，但 System Prompt 和对话历史中绝不出现。LLM 调用 `get_current_user_phone()` 工具时，工具返回"当前用户手机号已认证"而非明文号码。

**Q6: 地址修改这类敏感操作如何保障安全？**

双层保护：一是工具层验证订单归属（`order.user_id == user.id`）；二是用户确认机制。修改地址时，Agent 调工具写入变更并返回标记"申请编号：ADDR..."，主图检测到标记后路由到 `human_approval` 节点，`interrupt()` 暂停图执行，前端弹出确认横幅。用户点取消时，节点读取 approval_meta 中的旧地址，调用 `update_order_address_sync` 回滚。

### 工程实现

**Q7: SSE 流式输出中如何处理 interrupt 暂停导致的连接阻塞？**

`interrupt()` 会阻塞 `graph.stream()` 迭代器，导致后面的 `get_state()` 检查永远执行不到。我的解决方案是轮询机制：在 stream 线程之外，主循环使用 `queue.get(timeout=0.3)`，超时后主动调用 `get_state()` 检查是否有中断。发现中断立即通过 SSE 发送 `approval_required` 事件给前端，同时保持连接存活等待用户确认。

**Q8: PostgresSaver 的选型考虑？同步 vs 异步？**

最初考虑用 `AsyncPostgresSaver`，但在 Windows 上 psycopg3 的异步模式与 ProactorEventLoop 不兼容。最终使用同步版 `PostgresSaver` + `psycopg_pool.ConnectionPool`，在 FastAPI 异步端点中通过 `asyncio.to_thread()` 调用 graph 的同步方法。实测延迟增加不到 10ms，换取跨平台兼容性。

**Q9: DashScope Embeddings 遇到了什么问题？怎么解决的？**

新版 openai 库（2.45）发送的 `/v1/embeddings` 请求格式与 DashScope 原生 API 不兼容（input 字段格式差异），返回 400 错误。我自定义了 `DashScopeEmbeddings` 类继承 `langchain_core.embeddings.Embeddings`，用 httpx 直调 DashScope 原生 embedding endpoint，支持批量请求优化（每次最多 10 条文本，减少网络往返）。

**Q10: 如何评估系统质量？**

使用 LangSmith 平台，编写了 30 条测试用例覆盖全部意图（订单/商品/售后/FAQ/转人工/寒暄），5 个评估维度：意图识别准确率、关键词覆盖率、LLM-as-Judge 综合质量、结构完整性、长度合理性。每次提交后运行评估对比分数变化，发现并修复了转人工路由崩溃、FAQ 正则缺失、LLM 改写工具输出导致确认弹窗不触发等关键问题。

### 场景与边界

**Q11: 如果用户未登录就要求查订单，系统如何处理？**

预路由或意图分类器会将意图设为 `order`，Supervisor 调度到订单 Agent。订单 Agent 调用 `list_my_orders` 工具，工具从 config 中读到 `user_phone` 为空，返回"您还未登录，请先输入手机号"。Agent 将此消息返回给用户。整个过程不抛异常，不会泄露其他用户的订单数据。

**Q12: 如果 LLM 返回了不在预期内的路由决策怎么办？**

Supervisor 有兜底逻辑：LLM 输出的决策如果不属于合法列表（order/product/aftersale/faq/FINISH/human），则回退到默认意图。同时设置了安全上限（最多 3 轮调度），防止死循环。

**Q13: 不同用户在同一设备上先后登录，记忆会混合吗？**

不会。前端 `doLogout()` 调用 `newSession()` 生成新的 session_id，旧会话的 checkpoint 保持不变。`prepare_context_node` 在每次注入前用 `RemoveMessage` 清除旧的 `[系统记忆]` 消息。用户画像通过手机号 SHA256 哈希隔离存储在不同行。

**Q14: 系统如何扩展到 100 个 Agent？**

当前 Agent 构建模板已高度统一（60 行的 ReAct 子图样板可通过工厂函数消除）。增加新 Agent 只需：定义 System Prompt + 工具列表 → 调用 `build_agent()` 工厂 → 在主图 `AGENT_NAMES` 列表注册 → Supervisor 自动识别。正则预路由规则和 Supervisor LLM 提示词需同步更新。

**Q15: 如果 DashScope API 超时或限流，用户看到什么？**

`_invoke_subgraph` 有 try/except 兜底，捕获异常后返回"AIMessage：抱歉，系统处理您的请求时遇到问题，请稍后重试。如需紧急帮助请拨打 400-888-8888。"同时在后端打印异常日志。不会向用户暴露原始错误信息或堆栈。

### MCP 与扩展

**Q16: MCP 在你的项目中是如何落地的？**

FAQ Agent 本地知识库只覆盖 43 条预设问答。当用户问题超出范围时（如"2026 年电商法新规"），Agent 调用 `mcp_web_search` 工具，通过 DuckDuckGo Instant Answer API（免费无需 Key）搜索网页获取答案。工具采用双模设计：独立模式下直接 `import` 函数调用；安装 `mcp` 包后自动切换为标准 MCP stdio 协议。所有搜索记录写入 JSONL 日志便于分析。核心价值：Agent 的能力边界不再受限于本地代码仓库，新增工具只需在 MCP Server 目录添加文件并注册。

**Q17: MCP Server 的双模设计是怎么实现的？**

每个 MCP Server 文件底部有判断逻辑：`__name__ == "__main__"` 时尝试 `from mcp.server import Server`——如果 mcp 包已安装，注册为标准 MCP tool 并通过 stdio 运行；否则回退到独立模式直接调用函数。`get_langchain_mcp_tools()` 直接将底层异步函数包装为 LangChain `@tool` 装饰器，处理事件循环兼容（同步 Agent 中通过 `asyncio.run` 或线程池执行）。

### 多模态

**Q18: 图片上传功能如何与现有 Agent 系统集成？**

最小侵入原则：多模态只在消息入口做一次转换。用户上传图片 → `qwen-vl-plus` 视觉 LLM 分析内容 → 生成文本描述 → 作为 HumanMessage 注入消息流 → 后续意图识别和 Agent 路由完全不变。分析文本对用户透明：前端只显示图片缩略图，SSE token 过滤器和历史记录接口同时拦截 `[用户上传` 前缀的消息，用户感知不到分析中间产物。

**Q19: 如果要用真正的多模态模型替代当前的图片→文本方案，架构应该如何调整？**

将 `HumanMessage` 的内容从纯文本改为多模态列表 `[{"type":"text","text":"..."}, {"type":"image_url","image_url":{"url":"..."}}]`。关键改动点：(1) Agent 的 LLM 实例从 `qwen-plus` 切换为 `qwen-vl-plus`；(2) `_invoke_subgraph` 传入子图的消息需要保留多模态结构，不能只传纯文本；(3) 前端流式渲染不需要改——LLM 输出的仍是文本 token。当前架构中 Agent 子图与视觉处理完全解耦，切换成本主要在 LLM 配置层。
