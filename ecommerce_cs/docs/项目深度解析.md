## 1. 一次用户请求的完整旅程

以用户说 "帮我查一下我的订单" 为例，完整链路如下：

```
浏览器                                        服务端
─────                                        ─────
1. 用户输入文字，点击发送
2. JS: sendMessage()
   → EventSource 连接 /api/chat/{sid}/stream
   → 带上 message=...&phone=13800001001
                                             3. FastAPI: chat_stream()
                                             4. _get_config(session_id, phone_hint)
                                                → 查 session_phones 表拿 phone
                                                → 构建 config = {"configurable":
                                                   {"thread_id":..., "user_phone":"13800001001"}}
                                             5. _make_initial_state(message, session_id)
                                                → {"messages":[HumanMessage("帮我查...")], ...}
                                             6. _csr_graph.stream(initial_state, config)
                                                 ↓
                                             7. [graph.py] pre_router_node
                                                正则 "订单|我的.*单" 匹配 → intent="order"
                                                 ↓
                                             8. [graph.py] prepare_context_node
                                                从 DB 加载用户画像+摘要 → 注入 SystemMessage
                                                 ↓
                                             9. [graph.py] supervisor_node (第1轮)
                                                LLM 决策 → "order"
                                                 ↓
                                             10. [graph.py] call_order_agent
                                                  → _invoke_subgraph("order", state, config)
                                                  → 启动 order_agent 子图
                                                     ↓
                                             11. [agents/order_agent.py] agent_node
                                                 System Prompt + 消息 → LLM 决定调哪个工具
                                                 → 调 list_my_orders()
                                                     ↓
                                             12. [tools/order_tools.py] list_my_orders(config)
                                                 从 config 读 user_phone → 查 DB → 返回订单列表
                                                     ↓
                                             13. LLM 将工具返回值整理成自然语言
                                             14. 子图结束，返回最后一条 AI 消息
                                                 ↓
                                             15. [graph.py] supervisor_node (第2轮)
                                                 自动 FINISH → END
                                             16. SSE 流式返回 token 给前端
17. JS: handleSSEEvent()
    → "token" 事件逐字追加到气泡
    → "done" 事件隐藏状态栏
    → 后台触发 /compress 压缩记忆
```

关键点：
- 整个链路是同步阻塞的（`graph.stream()` 在线程中运行），但通过 `asyncio.to_thread` 不阻塞事件循环
- 每个节点只修改 state 中自己关心的字段，LangGraph 负责合并
- config 像一根线穿过所有节点、子图、工具函数，携带 thread_id 和 user_phone

## 3. 核心架构

### 3.1 Supervisor 模式

```
                    START
                      │
              pre_router_node (正则盲筛)
                      │
            prepare_context_node (记忆注入)
                      │
                 ┌────┴────┐
          命中正则   未命中
             │         │
             │    intent_classifier (LLM)
             │         │
             └────┬────┘
                  │
            supervisor_node (LLM 动态路由)
                  │
       ┌──────────┼──────────┬──────────┐
       │          │          │          │
  order_agent  product   aftersale   faq_agent
       │          │          │          │
       └──────────┴──────────┴──────────┘
                  │
            supervisor_node (第 2 轮自动 FINISH)
                  │
                 END
```

Supervisor 只做一件事：根据对话上下文决定"下一步找谁"。

- **第 1 轮**：LLM 根据意图 + 用户消息决定路由
- **第 2 轮起**：系统级自动 FINISH（子 Agent 的 ReAct 循环已经处理完毕）
- **安全上限**：最多 3 轮，防止死循环
- **不再依赖 [DONE] 文本标记**：旧版在 prompt 里要求 LLM 输出 [DONE] 表示结束，存在 prompt 注入风险

### 3.2 为什么用子图而非一张大图

一张大图的问题：所有 Agent 共享同一个 ReAct 循环，System Prompt 和工具列表无法隔离。子图的好处：

1. **隔离**：每个子图有独立的 System Prompt + 独立 LLM 实例 + 独立工具集
2. **可维护**：新增 Agent = 新增文件 + 注册节点，不影响现有 Agent
3. **debug**：子图内部的消息流不会污染父图的消息历史

子图编译时只返回 `StateGraph`（不传 checkpointer），由父图注入。这意味着子图是无状态的——它们依赖父图的 checkpointer 来持久化。

## 4. 数据流

### 4.1 State：整个图共享的数据字典

```python
class CSRState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # 对话历史（自动追加合并）
    intent: str              # 当前意图
    iteration_count: int     # Supervisor 调度轮数
    next_agent: str          # Supervisor 决定的下一个 Agent
    user_phone: str          # 当前用户手机号（从 config 同步）
    summary: str             # 历史摘要
    user_profile_json: str   # 用户画像
    approval_decision: str   # 审批决策（approve/reject）
    approval_meta: str       # 审批元数据（回滚信息）
```

State 通过节点返回值增量更新。例如 `call_order_agent` 返回 `{"messages": [AIMessage(...)]}` 时，只有 messages 字段被更新（通过 `add_messages` reducer 追加而非覆盖）。

### 4.2 Config：跨层传递的上下文

```python
config = {
    "configurable": {
        "thread_id": "sess_xxx",     # 会话 ID，隔离不同用户
        "user_phone": "13800001001"  # 手机号，工具函数读取
    }
}
```

传递链路：
```
server.py: _get_config() 构建 config
  → graph.stream(initial_state, config=config)
  → 各节点函数(state, config) 接收
  → 子图 .invoke(state, config) 转发
  → agent_node: llm_with_tools.invoke(messages, config)
  → ToolNode 自动把 config 传给每个 @tool 函数
  → 工具函数: config["configurable"]["user_phone"] 读取手机号
```

关键点：手机号只存在于 config 中，不在 state.messages 里，所以 LLM 的 System Prompt 和对话历史中绝对看不到明文手机号。

### 4.3 State vs Config 的区别

| | State | Config |
|------|-------|--------|
| 持久化 | PostgresSaver 每次写 checkpoint | 不持久化，每次请求重建 |
| 传递方式 | 节点返回值合并 | 函数参数显式传递 |
| 用途 | 对话历史、意图、决策 | 会话 ID、用户身份 |
| 大小 | 可能很大（消息历史） | 很小（几个字段） |

## 5. 意图识别

### 5.1 双层架构

```
用户消息
  │
  ├── 第1层: pre_router_node（正则盲筛）
  │     命中 → 直接返回 intent，跳过 LLM
  │     未命中 → 返回 intent="__llm__"
  │
  └── 第2层: intent_classifier_node（LLM 分类）
        仅正则未命中时触发，LLM 分析消息输出意图标签
```

### 5.2 正则规则（ROUTE_RULES）

规则按顺序匹配，先命中先返回：

| 优先级 | 意图 | 正则关键词 |
|--------|------|-----------|
| 1 | order | 订单/物流/快递/发货/改地址/我的.*单 |
| 2 | aftersale | 退货/换货/退款/投诉/质量.*问题 |
| 3 | product | 推荐/有没有/多少钱/对比/选.*哪 |
| 4 | faq | 支付/花呗/会员/积分/营业时间/你好/谢谢/在吗 |
| 5 | human | 转人工/人工客服 |

覆盖约 80% 的常见问题，命中后秒级响应，不需要等待 LLM。

### 5.3 LLM 分类器（兜底）

正则未命中时，将消息发送给 LLM 分类。LLM 输出单个标签（order/product/aftersale/faq/human），有兜底逻辑（非法输出 → faq）。

## 6. 子 Agent

### 6.1 四个 Agent 的结构完全相同

每个 Agent 文件包含：
1. `XxxAgentState(TypedDict)` — 只含 messages 字段
2. `build_xxx_agent()` — 构建 StateGraph
3. `agent_node(state, config)` — LLM 推理节点
4. `SYSTEM_PROMPT` — 角色 + 工具清单 + 行为规则

### 6.2 ReAct 循环

```
START → agent_node → tools_condition → ToolNode → agent_node → ... → END
```

- `agent_node`：System Prompt + 对话历史 → LLM → 输出 AIMessage（可能含 tool_calls）
- `tools_condition`：LangGraph 内置函数，判断 LLM 输出中是否有 tool_calls
- `ToolNode`：执行工具调用，生成 ToolMessage
- 循环直到 LLM 不再调用工具（输出纯文本 AIMessage）

### 6.3 子图的工具集

| Agent | 自有工具 | 共享工具 |
|-------|---------|---------|
| order_agent | query_order / track_shipment / list_my_orders / modify_shipping_address | get_current_user_phone |
| product_agent | search_products / get_product_detail / check_stock / search_product_knowledge_tool | get_current_user_phone |
| aftersale_agent | check_return_policy / create_return_request / query_return_status + list_my_orders + query_order | get_current_user_phone |
| faq_agent | search_faq / get_faq_categories | get_current_user_phone |

aftersale_agent 特殊：额外导入了订单工具，因为售后流程需要先列订单再让用户选。

## 7. RAG 知识库

### 7.1 五层知识结构

```
Chroma 向量库
├── 商品详情文档（204 篇）—— 每件商品一篇，含参数/场景/选购建议
├── 品类选购指南（8 篇）—— 手机/电脑/家电/服饰/食品/美妆/运动/家居
├── 售后政策详解（1 篇）—— 退货/换货/退款/价保规则
├── FAQ 知识（43 条）—— 配送/支付/会员/账号安全/购物指南
└── 自定义文档 —— 通过知识库管理页面添加
```

每篇文档用 `RecursiveCharacterTextSplitter(chunk_size=500, overlap=80)` 切分为文本块，总计 463 个 chunk。

### 7.2 嵌入模型

`DashScopeEmbeddings` 自定义类（[config.py](config.py#L98)）：因为 openai 2.45 客户端与 DashScope embedding API 输入格式不兼容，直接用 httpx 调原生 API。支持批量请求（每次 10 条）。

### 7.3 检索流程

```
用户问 "降噪耳机怎么选"
  → search_product_knowledge_tool("降噪耳机选购")
  → get_product_retriever(k=4).invoke(query)
  → Chroma.similarity_search(query, k=4)
  → 返回 4 个最相关的文档片段
  → 格式化为带标签的文本返回给 LLM
```

### 7.4 知识库管理（端口 8001）

独立 FastAPI 服务，通过 langchain_community 的 Chroma 包装器操作底层 collection：
- CRUD：新增/编辑/删除文档（长文档自动切分）
- 搜索：语义相似度检索
- 统计：分类和类型分布
- 重建：从 PostgreSQL 重新生成全部文档并构建索引

## 8. 记忆系统

### 8.1 三层架构

```
第 1 层：滑动窗口（Short-Term Memory）
  └── 最近 6-12 条 human/ai 消息原样保留在对话中

第 2 层：摘要压缩（Summary）
  └── 超出窗口的历史 → LLM 压缩为 150 字情景摘要 → 存入 PostgreSQL user_profiles.summary

第 3 层：用户画像（Profile）
  └── 从对话中抽取偏好/事实 → JSON 存入 PostgreSQL user_profiles
```

### 8.2 压缩时机（异步，不阻塞主流程）

```
用户在浏览器看到 AI 回复
  → SSE 发送 "done" 事件
  → 前端自动调用 POST /api/chat/{session_id}/compress
  → 后端 asyncio.to_thread 运行 compress_and_save()
  → LLM 压缩历史 + 抽取画像 → 写入 PostgreSQL
  → 用户完全感知不到
```

### 8.3 上下文注入

每轮对话开始时（`prepare_context_node`），从 PostgreSQL 读取当前用户的摘要和画像，注入为 SystemMessage：

```
[系统记忆]
【历史对话摘要】用户上次询问了订单 ORDxxx，购买了 AirPods Pro...
【用户画像】
偏好：降噪耳机、关注性价比
信息：城市:北京
```

不同手机号 → 不同 user_id（SHA256 哈希）→ 不同数据库行 → 物理隔离。

## 9. 流式输出

### 9.1 整体架构

```
graph.stream() 在后台线程运行
  → stream_mode=["messages","updates"]
  → subgraphs=True（子图事件也流式输出）
  → 事件入队列 queue.Queue()
  → 主循环读队列 → 过滤 → SSE yield
```

### 9.2 Token 过滤规则

不是所有 AI 消息都发给前端。当前白名单：

1. `node_name == "agent"` — 只放行子 Agent 内部 LLM 节点的输出
2. 过滤 `tool_calls` / `tool_call_chunks` — 工具调用参数的 JSON 碎片不显示
3. 过滤 `[系统记忆]` 开头的内容 — 记忆注入消息不显示
4. 过滤纯意图标签（order/product/aftersale/faq/human/FINISH）

### 9.3 interrupt 轮询机制

`interrupt()` 会阻塞 `graph.stream()` 的 for 循环。轮询方案：

```
主循环: while True:
    try:
        item = q.get(timeout=0.3)   # 0.3 秒超时
    except Empty:
        if stream_active:
            state = get_state()     # 主动检查中断
            if 有中断:
                → SSE: approval_required
                → 前端显示确认横幅
                break
    # 正常 stream 事件处理...
```

用户点击确认/取消后，`approve_action/reject_action` 端点调用 `Command(resume="approve"/"reject")` 恢复图执行。

## 10. 安全设计

### 10.1 手机号隔离

```
用户输入手机号 → POST /api/login/{sid}
  → 查 users 表验证 → 写入 session_phones 表
  → config["configurable"]["user_phone"] = phone
  → 工具从 config 读取，System Prompt 中不出现
  → LLM 不知道用户手机号
```

### 10.2 订单/售后归属验证

每个涉及数据的工具都验证归属权：

```python
def query_order(order_no, config):
    phone = config.get("configurable", {}).get("user_phone", "")
    user = find_user_by_phone(phone)
    order = find_order_by_no(order_no)
    if order.user_id != user.id:
        return "此订单不属于您"
    # ...返回订单详情
```

### 10.3 用户确认 + 回滚

地址修改和售后申请需要用户在弹窗中确认：

```
Agent 调 modify_shipping_address
  → 工具写入新地址，返回 "申请编号：ADDR..."
  → call_order_agent 检测到 "申请编号"
  → 路由到 human_approval_node
  → interrupt() 暂停
  → 前端弹窗 "请确认此操作"
  → 用户点确认：地址保持
  → 用户点取消：读取 approval_meta 中的旧地址 → 回滚
```

## 11. 会话持久化

### 11.1 PostgresSaver

LangGraph 的 checkpointer 机制：每次图执行到一个节点，自动保存 state 快照到 PostgreSQL。

三张 checkpoint 表：
- `checkpoints`：主快照（state 序列化 + metadata）
- `checkpoint_blobs`：大字段（消息内容等）
- `checkpoint_writes`：待处理的写入

### 11.2 session_phones 表

```sql
CREATE TABLE session_phones (
    session_id VARCHAR(64) PRIMARY KEY,
    phone VARCHAR(20) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);
```

替代了早期纯内存的 `_session_phones` dict，服务器重启后会话绑定不丢失。

### 11.3 启动清理

每次启动时自动清理无 phone 绑定的孤立 checkpoint：

```sql
DELETE FROM checkpoints WHERE thread_id IN (
    SELECT c.thread_id FROM checkpoints c
    LEFT JOIN session_phones sp ON sp.session_id = c.thread_id
    WHERE sp.session_id IS NULL
);
```

## 12. 关键踩坑记录

### 12.1 DashScope Embeddings API 不兼容

**现象**：openai 2.45 客户端请求 `/v1/embeddings` 返回 400。

**原因**：openai 库的请求格式（`input` 字段结构）与 DashScope 原生 API 不一致。

**解决**：自定义 `DashScopeEmbeddings` 类，用 httpx 直调原生 endpoint `https://dashscope.aliyuncs.com/api/v1/services/embeddings/text-embedding/text-embedding`。

### 12.2 Windows GBK 编码乱码

**现象**：`print()` 输出中文日志报 `UnicodeEncodeError`。

**解决**：`start_all.py` 子进程设置 `PYTHONUTF8=1` + `PYTHONIOENCODING=utf-8`。`graph.py` 的 `_safe_print` 三层兜底：正常 print → `sys.stdout.buffer.write(utf-8)` → ASCII fallback。

### 12.3 interrupt_before 与 subgraphs 流式兼容性

**现象**：`interrupt_before=["human_approval"]` 在 `subgraphs=True` 时行为不稳定，有时不暂停。

**解决**：改为节点内 `interrupt()` 函数调用 + SSE 轮询机制，而非依赖编译参数。

### 12.4 LLM 改写工具输出导致确认弹窗不触发

**现象**：工具返回 "申请编号：RTN..."，但 LLM 在回复中改写为 "工单号为"，`call_aftersale_agent` 的字符串匹配失败。

**解决**：遍历子图返回的**全部消息**（含 ToolMessage），不只检查最后一条 AI 消息。`_has_approval_marker()` 函数搜索所有消息中的 "申请编号"。

### 12.5 知识库重建缓存未清除

**现象**：POST /api/rebuild 后，搜索端点仍返回旧数据。

**原因**：`rebuild_index` 重置了不存在的 `_collection`，实际缓存变量是 `_vectorstore`。

**解决**：修正为 `global _vectorstore; _vectorstore = None`。

### 12.6 转人工路由 key 不匹配

**现象**：用户说"转人工"后报错 `ERROR: 'human_handoff'`。

**原因**：`route_by_next_agent` 返回 `"human_handoff"`（节点名），但边映射的 key 是 `"human"`。

**解决**：统一为 `"human"`。

### 12.7 PostgresSaver Windows 兼容

**现象**：`AsyncPostgresSaver` 在 Windows 上报 ProactorEventLoop 错误。

**解决**：使用同步版 `PostgresSaver` + `psycopg_pool.ConnectionPool`，FastAPI 端点通过 `asyncio.to_thread()` 调用。

## 13. MCP 扩展

### 13.1 什么场景需要 MCP

FAQ Agent 的知识库只覆盖了 43 条预设问答。当用户问"2026 年电商法有什么新规"这类不在 FAQ 中的问题，Agent 无法回答。MCP 让 Agent 可以调用外部工具——在此例中，自动上网搜索。

### 13.2 当前挂载的 MCP 工具

| 工具 | 来源 | 触发条件 |
|------|------|---------|
| `mcp_web_search(query)` | DuckDuckGo Instant Answer API | FAQ 搜索无结果时 |
| `mcp_create_ticket(title, priority)` | 本地 JSONL 日志 | 需要人工跟进时 |

### 13.3 实现方式

双模设计：MCP Server 中的函数既可以通过 `import` 直接调用（独立模式），也可以通过 `mcp` 包的 stdio 协议作为标准 MCP Server 运行。

```
mcp_config.py
  └── get_langchain_mcp_tools()
        ├── 从 mcp_servers/web_search/server.py import web_search
        ├── 从 mcp_servers/demo_notify/server.py import create_ticket
        ├── 包装为 LangChain @tool 装饰器
        └── 返回 Tool 列表 → Agent 的 llm.bind_tools() 绑定
```

FAQ Agent 在 `build_faq_agent()` 中调用 `get_langchain_mcp_tools()`，将返回的工具合并到 `_AGENT_TOOLS` 中。LLM 在推理时自动决定何时调用这些工具。

### 13.4 调用链

```
用户："2026年双十一什么时候"
  → FAQ Agent: search_faq("双十一") → 无结果
  → FAQ Agent: mcp_web_search("2026年双十一开始时间")
  → DuckDuckGo API → 返回搜索结果
  → FAQ Agent: 整理搜索结果 → "根据最新信息，2026年双十一预计..."
```

所有搜索日志记录在 `mcp_servers/web_search/logs/search.jsonl`。

## 14. 多模态入口

### 14.1 设计原则

多模态只在消息入口做一次转换，不侵入 Agent 逻辑。图片 → 视觉 LLM → 文本描述 → HumanMessage → 现有 Agent 链路完全复用。

### 14.2 实现

```
POST /api/upload/{session_id}
  1. 保存图片到 uploads/{session_id}/ 目录
  2. qwen-vl-plus 视觉 LLM 分析图片内容
  3. 返回分析文本

前端 uploadImage():
  1. 显示图片缩略图
  2. POST /api/upload → 拿到分析文本
  3. 将 "[用户上传了一张图片，图片内容：{分析}]" 作为 HumanMessage 发送
  4. SSE 流式接收 Agent 回复
  5. 分析文本不显示在前端（被 SSE token 过滤器和历史记录过滤器拦截）
```

### 14.3 支持的输入方式

| 方式 | 前端实现 |
|------|---------|
| 粘贴图片 | `chatInput.onpaste` → `handlePaste()` |
| 拖拽图片 | `chatMessages.ondrop` → `uploadImage()` |
| 选择文件 | 输入框旁的图片按钮 → `handleImageSelect()` |

### 14.4 视觉 LLM 的 prompt

```
"你是一个电商客服助手。请简短描述这张图片的内容。
如果是订单截图，提取订单号。如果是商品照片，描述外观和可能的问题。
如果是聊天记录，提取关键信息。2-3句话。"
```
